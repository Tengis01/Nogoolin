## 1. Товч Тойм

**Ногоолин** нь Монголын зах зээлд зориулан бүтээгдсэн шашны бүтээгдэхүүний premium каталог болон ирээдүйд хүргэлт хийх боломжтой дижитал платформ юм.

Платформ нь хэрэглэгчдэд уламжлалт шашны болон оюун санааны бүтээгдэхүүнийг — бурхны дүрс, хүж, тахилын зүйл, залбирлын хэрэгсэл болон бусад зүйлсийг — онлайнаар олж мэдэх, судлах, ирээдүйд захиалах боломжийг олгоно. Энэ төрлийн бүтээгдэхүүн нь одоогоор зөвхөн физик дэлгүүрээр худалдагдаж байгаа бөгөөд орчин үеийн дижитал туршлага огт байхгүй байна.

Төслийг нэг хөгжүүлэгч бие даан хийж байгаа бөгөөд энэ нь бодит бизнесийн бүтээгдэхүүн төдийгүй, мэргэжлийн portfolio төсөл болох зорилготой. Эхний мөрөнд кодлохоос өмнө бүрэн баримт бичиг, зохитой архитектуртай байхаар стандартаар бүтээгдэж байна.

## 2. Асуудлын Тодорхойлолт

### 2.1 Одоогийн Байдал

Шашны болон оюун санааны бүтээгдэхүүн нь Монголчуудын өдөр тутмын амьдралд онцгой соёлын ач холбогдолтой. Гэвч одоогийн худалдааны туршлага тархай бутархай, хүртээмжгүй байна:

- Бүтээгдэхүүнүүд Улаанбаатарын тархай бутархай физик дэлгүүрүүдэд зарагдаж байгаа бөгөөд онлайн нэгдсэн платформ байхгүй
- Хэрэглэгчид бүтээгдэхүүнийг хайж олох, харьцуулах, судлах зориулалттай дижитал каталог байхгүй
- Бүтээгдэхүүний хэрэглэх заавар хаана ч бичигдэж тэмдэглэгдэж байхгүй — худалдан авагчид аман дамжуулалт эсвэл дэлгүүрт очиж асуухаас өөр аргагүй
- Эдгээр бүтээгдэхүүний нандин шашны шинж чанарт нийцсэн мобайл апп эсвэл вэб туршлага огт байхгүй

### 2.2 Зөрүү

| Өнөөдөр байгаа зүйл | Дутагдаж байгаа зүйл |
|---|---|
| Зөвхөн физик дэлгүүрүүд | Онлайн бүтээгдэхүүний каталог |
| Дижитал нээлт байхгүй | Ангилал, хайлт, шүүлтүүртэй үзэх боломж |
| Онлайн зааварчилгаа байхгүй | Алхам алхмаар хэрэглэх заавар |
| Premium брэндийн туршлага байхгүй | Утга учиртай, соёлоо хүндэтгэсэн визуал дизайн |
| Захиалах боломж байхгүй | Хүргэлтийн дэд бүтэц |

### 2.3 Яагаад Одоо

Монголд ухаалаг утас болон мобайл интернетийн хэрэглээ хурдацтай нэмэгдэж байна. Хэрэглэгчид бараа бүтээгдэхүүнийг худалдан авахаасаа өмнө онлайнаар хайж судлахыг улам бүр хүлээж байна. Одоо чанартай дижитал туршлага бүтээх нь энэ өргөн тэлж буй ништ анхдагч байрлалыг тогтооно.

## 3. Хэрэглэгчид

### 3.1 Үндсэн Хэрэглэгч — Каталогийн Зочин (1-р Шат)

**Хэн бэ:**  
Шашны болон оюун санааны бүтээгдэхүүн тогтмол худалдан авдаг буюу сонирхдог Монгол насанд хүрэгчид — голдуу 25–60 насны эмэгтэйчүүд. Тэд туршлагатай худалдан авагч байж болох бөгөөд эсвэл энэ чиглэлд шинээр орж байгаа хүн байж болно.

**Юу хэрэгтэй вэ:**
- Цэвэр, итгэлтэй интерфэйсээр ангилал тус бүрийн бүтээгдэхүүнийг үзэх
- Олон өнцгөөс авсан өндөр чанарын зураг харах
- Бүтээгдэхүүний дэлгэрэнгүй тайлбар болон хэрэглэх зааврыг унших
- Дэлгүүрт очиж үзэх эсвэл хүсэлт гаргахаасаа өмнө үнийг мэдэх
- Гэр бүлийнхэндээ бүтээгдэхүүний холбоосыг хуваалцах

**Өвдөлтийн цэгүүд:**
- Онлайнаар найдвартай мэдээлэл олох боломжгүй
- Зарим тахилын зүйлийг хэрхэн зөв хэрэглэхийг мэдэхгүй
- Олон дэлгүүр зорчилгүйгээр бүтээгдэхүүнийг харьцуулах боломжгүй

---

### 3.2 Дараалах Хэрэглэгч — Хүсэлт Гаргагч (4-р Шат)

**Хэн бэ:**  
Каталогийн зочин бүтээгдэхүүн авахаар шийдсэн боловч бүрэн захиалга хийхийн өмнө сонирхлоо илэрхийлэх буюу асуулт тавихыг хүсч байгаа хэрэглэгч.

**Юу хэрэгтэй вэ:**
- Бүрэн бүртгэлгүйгээр хялбар хүсэлт гаргах
- Хүсэлт хүлээн авагдсан гэдгийн баталгаа
- Бизнесийн талаас буцаан холбоо барих

---

### 3.3 Ирээдүйн Хэрэглэгч — Захиалагч (5-р Шат+)

**Хэн бэ:**  
Бүрэн захиалга хийж, хүргүүлэхэд бэлэн хэрэглэгч.

**Юу хэрэгтэй вэ:**
- Сагс болон төлбөрийн урсгал
- Захиалгын баталгаа болон статус хянах
- Хаягаар хүргэлт хийлгэх

> **Тэмдэглэл:** Энэ хэрэглэгч нь ирээдүйн persona юм. Туршлага нь архитектурын хувьд төлөвлөгдсөн боловч MVP-д идэвхжүүлэгдэхгүй.

---

### 3.4 Дотоод Хэрэглэгч — Администратор

**Хэн бэ:**  
Платформыг удирдаж байгаа бизнесийн эзэн буюу оператор.

**Юу хэрэгтэй вэ:**
- Бүтээгдэхүүний жагсаалтыг үүсгэх, засах, удирдах
- Бүтээгдэхүүний зураг байршуулж удирдах
- Ангиллуудыг удирдах
- Хэрэглэгчдийн хүсэлтийг харах, хариулах
- Үйл ажиллагаа бэлэн болоход хүргэлтийг асаах/унтраах
- Хийсэн өөрчлөлтүүдийг аудитлах

## 6. MVP Хамрах Хүрээ

MVP нь хөгжүүлэлтийн замын зургийн **1-р шатаас 3-р шат хүртэл** гэж тодорхойлогдоно.

### MVP-д Орох Зүйлс

**Нийтийн Вэбсайт (Next.js)**
- Premium 3D нээлтийн анимаци (Ногоон Дарь Эх, Three.js + React Three Fiber)
- Skip товч, статик fallback, reduced motion дэмжлэг
- Онцлох бүтээгдэхүүнтэй нүүр хуудас
- Бүтээгдэхүүний жагсаалтын хуудас (ангиллын шүүлтүүртэй)
- Бүтээгдэхүүний дэлгэрэнгүй хуудас (зураг, үнэ, тайлбар, хэрэглэх заавар)
- **360° интерактив 3D загварын харагч** бүтээгдэхүүний дэлгэрэнгүй хуудсанд (Three.js OrbitControls); GLB байхгүй бол зургийн gallery харуулна
- Ангиллаар үзэх
- Кирилл монгол, латин монгол, англи гурван бичгийн системд нэгэн зэрэг ажилладаг бүтээгдэхүүний хайлт
- Бүтээгдэхүүний хүсэлт/холбоо барих маягт
- SEO оновчлогдсон хуудсууд (meta title, description, Open Graph, цэвэр slug, sitemap)

**Администраторын Самбар (Next.js `/admin` route)**
- Администраторын аюулгүй нэвтрэлт (email/password + Google OAuth)
- Бүтээгдэхүүний CRUD (үүсгэх, засах, нийтлэх/архивлах, устгах)
- Ангиллын CRUD
- Бүтээгдэхүүний зураг байршуулах (Supabase Storage)
- **3D загварын файл (GLB) байршуулах** — Meshy AI workflow-оор үүсгэж, gltf-pipeline-аар compress хийсэн
- Хэрэглэх заавар удирдах
- Хүсэлтийн ирэлтийн хайрцаг болон статус удирдах
- Хүргэлтийн тохируулгын товч

**Мобайл Апп (Flutter)**
- Rive-д суурилсан нээлтийн анимаци
- Нүүр хуудас
- Ангиллын шүүлтүүртэй бүтээгдэхүүний жагсаалт
- Бүтээгдэхүүний дэлгэрэнгүй хуудас
- **360° интерактив 3D загварын харагч** `model_viewer_plus` ашиглан; GLB байхгүй бол зургийн gallery харуулна
- Бүтээгдэхүүний хүсэлтийн маягт

**Backend API (Express.js + TypeScript)**
- REST API `/api/v1/` хаягаар
- Давхаргат архитектур: Controller → Service → Repository
- Supabase PostgreSQL + Supabase Auth + Supabase Storage
- Дүрд суурилсан хандалт удирдлагатай JWT баталгаажуулалт (admin / customer)
- 4 давхаргат аюулгүй байдал: Cloudflare WAF → Helmet/CORS/rate-limit → JWT/RBAC → RLS

### MVP-д Орохгүй Зүйлс

Архитектурын хувьд **төлөвлөгдсөн** боловч MVP-д **идэвхжүүлэгдэхгүй** зүйлс:

- Сагс болон төлбөр хийх
- Захиалга хийх
- Хүргэгчийн удирдлага
- Төлбөрийн интеграци
- Захиалга хянах

## 7. Хийхгүй Зүйлс

Дараах зүйлсийг **хамрах хүрээнд** орохгүй бөгөөд ойрын ирээдүйд **бүтээхгүй**:

| Хийхгүй зүйл | Шалтгаан |
|---|---|
| Олон худалдагчтай зах зээл | Энэ нь нэг бизнесийн платформ |
| Бодит цагийн чат эсвэл амьд дэмжлэг | Энэ шатанд хэт төвөгтэй, үнэ цэнэ бага |
| Төлбөрийн систем интеграци | Бизнесийн лиценз болон хүргэлтийн бэлэн байдлаас хамаарна |
| Microservices архитектур | Нэг хөгжүүлэгчид шаардлагагүй дэд бүтцийн ачаалал |
| Мобайлд захиалгат 3D renderer | `model_viewer_plus` хэрэгцээг хангана; Three.js/flutter_gl production-д бэлэн биш |
| Нарийн хүргэгчийн хянах апп | Зөвхөн ирээдүйн шат |
| Зургаар бүтээгдэхүүн хайх (камер → ижил төстэй бүтээгдэхүүн олох) | 6-р шат+ гэж төлөвлөгдсөн; pgvector + embedding pipeline шаардлагатай; каталог эхлээд дүүргэгдэх ёстой |
| Олон хэлний дэмжлэг | Монгол-даач; Англи хэл дараа нэмэгдэж болно |
| Хэрэглэгчийн контент / сэтгэгдэл | Ирээдүйн шат |
| Нийгмийн худалдааны функцүүд | Хамрах хүрээнд багтахгүй |

## 8. Амжилтын Хэмжүүрүүд

### 0–1-р Шатны Амжилт (Баримт бичиг + Суурь)
- [ ] Кодлохоос өмнө 0-р шатны бүх баримт бичгүүд дуусгагдсан байх
- [ ] Monorepo, CI/CD, Supabase schema ажиллаж байгаа байх
- [ ] Баталгаажуулалт ажиллаж байгаа байх (email + Google OAuth)
- [ ] Администраторын scaffold ажиллаж байгаа байх

### 2–3-р Шатны Амжилт (Бүтээгдэхүүний Каталог)
- [ ] Администратор 5 минутын дотор бүтээгдэхүүн үүсгэж, засаж, нийтлэх боломжтой байх
- [ ] Бүтээгдэхүүний жагсаалтын хуудас Монголын дундаж мобайл холболтод 2 секундын дотор ачаалагдах
- [ ] Бүтээгдэхүүний дэлгэрэнгүй хуудас Lighthouse SEO оноо ≥ 90 байх

- [ ] Администратор dashboard-аас бүх хүсэлтийг харж, хариулах боломжтой байх

## 10. Бүтээгдэхүүний Урт Хугацааны Хөгжил

Платформ дараах чиглэлд хөгжихөөр төлөвлөгдсөн:

```
1–3-р Шат (MVP)
  Бүтээгдэхүүний каталог + 3D нээлт + 360° бүтээгдэхүүний харагч + Администраторын самбар
    ↓
4-р Шат
  Хүсэлт / зөөлөн захиалгын систем
  (Хэрэглэгчид бүрэн төлбөр хийхгүйгээр сонирхлоо илэрхийлж чадна)
    ↓
5-р Шат
  Хүргэлтийн тохируулга идэвхжүүлэгдэнэ
  Бүрэн сагс → төлбөр → захиалга → хүргэлтийн урсгал
    ↓
6-р Шат+
  Төлбөрийн систем интеграци
  Захиалга хянах
  Хүргэгчийн мобайл апп
  Зургаар бүтээгдэхүүн хайх (камер → pgvector ашиглан ижил төстэй бүтээгдэхүүн олох)
    ↓
Ирээдүй
  Бүтээгдэхүүний каталог өргөжих
  Байнгын хэрэглэгчийн онцгой эрхийн функцүүд
  Боломжит төлбөрийн хэтэвч интеграци
```

Энэ хөгжлийг боломжтой болгодог үндсэн архитектурын зарчим:

> **Хүргэлтийн систем нь мэдээллийн сангийн схем болон API-д бүрэн төлөвлөгдөж, бүтээгдсэн боловч бизнесийн үйл ажиллагаа бэлэн болтол администраторын удирдамжтай тохируулгын ард идэвхгүй хэвээр байна.**

Энэ нь хүргэлтийг идэвхжүүлэхийн тулд томоохон өөрчлөлт хийх шаардлагагүй гэсэн үг — зөвхөн тохируулга өөрчлөгдөнө.

## 11. Төслийн Зарчмууд

### Энэ Төсөл Яагаад Байгаа Вэ

Ногоолин нэгэн зэрэг хоёр зорилго биелүүлнэ:

1. **Бодит бүтээгдэхүүн** — Бодит Монгол хэрэглэгчдэд үйлчилдэг бодит бизнесийн ажиллагаатай платформ
2. **Portfolio төсөл** — Баримт бичиг-эхлэлт хөгжүүлэлт, зохитой архитектур, CI/CD, аюулгүй байдлын дизайн, вэб/мобайл/backend дамнасан бүрэн stack хэрэгжүүлэлтийг агуулсан production-чанарын програм хангамжийн инженерчлэлийн дадлагыг харуулах

### Инженерчлэлийн Философи

- **Кодоос өмнө баримт бичиг.** Хэрэгжүүлэлт эхлэхээс өмнө бүх архитектур, шаардлага, хэрэглэх тохиолдол, API spec бичигдсэн байна.
- **Эхний өдрөөсөө зөв архитектур.** Concerns-ийн цэвэр тусгаарлалттай давхаргат monolith — яаравчилсан прототип биш.
- **Загвараасаа л аюулгүй байдал.** Backend-ийн эхний код бичигдэхээс өмнө 4 давхаргат defense-in-depth загварчлагдсан байна.
- **Нэг хөгжүүлэгчид тохирсон практик сонголтууд.** Технологийн шийдвэр бүр нэг хөгжүүлэгчийн хязгаарлалтад үндэслэн зөвтгөгдсөн — хэт зохиомжлол байхгүй.
- **Тэмдэглэлгүйгээр ажиллагаатай програм хангамж.** Scrum-ийн оронд Personal Kanban. Төлөвлөлтөөс илүү хүргэлт.
